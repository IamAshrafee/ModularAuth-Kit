## Authentication (OWASP A07:2025)

| Requirement | Status | Our Implementation |
|---|---|---|
| Do not ship with default credentials | ✅ | No default users or passwords. All secrets from env vars. |
| Implement weak password checks | ✅ | Configurable password policy (min length, complexity rules via Zod) |
| Align password length/complexity to NIST 800-63b | ✅ | Min 8 chars (configurable), max 128 chars, no arbitrary complexity rules forced |
| Ensure registration, credential recovery, and API pathways are hardened against account enumeration | ✅ | Login: same error for "not found" and "wrong password". Forgot-password: same response whether user exists or not. |
| Limit failed login attempts | ✅ | Rate limiting per IP + optional account lockout after N failures |
| Use server-side session manager generating secure session IDs | ✅ | Custom server-side sessions, 256-bit entropy session IDs |
| Do not send session IDs in URLs | ✅ | Session ID only in HttpOnly cookie, never in URL |
| Invalidate sessions after logout, idle, and absolute timeouts | ✅ | Logout: delete session. Idle timeout: configurable. Absolute timeout: via expiresAt. |

## Session Management

| Requirement | Status | Our Implementation |
|---|---|---|
| Session ID has high entropy (128+ bits) | ✅ | 256 bits (32 bytes = 64 hex chars) via `crypto.randomBytes(32)` |
| Session ID generated server-side only | ✅ | Generated by `sessionService.create()`, never from client input |
| Session cookie uses HttpOnly flag | ✅ | Always `httpOnly: true` — JS cannot access |
| Session cookie uses Secure flag | ✅ | `secure: true` in production (HTTPS only) |
| Session cookie uses SameSite attribute | ✅ | `sameSite: 'lax'` by default (CSRF protection) |
| Session cookie uses non-default name | ✅ | Default: `"sid"` (not `"connect.sid"` or `"JSESSIONID"`) |
| Session ID rotated after authentication | ✅ | New session ID on login (`rotateOnLogin: true`) |
| Session invalidated on logout | ✅ | Session document deleted from DB, cookie cleared |
| Idle timeout implemented | ✅ | `lastActiveAt` tracked, checked on every validation |
| Absolute timeout implemented | ✅ | `expiresAt` set at creation, TTL index auto-deletes |
| Users can view and revoke active sessions | ✅ | `GET /auth/sessions`, `DELETE /auth/sessions/:id` (when feature enabled) |

## Password Storage

| Requirement | Status | Our Implementation |
|---|---|---|
| Use a strong adaptive hashing function | ✅ | argon2id — OWASP #1 recommendation |
| Use OWASP-recommended parameters | ✅ | Memory: 19 MiB, Iterations: 2, Parallelism: 1 |
| Salt is unique per password | ✅ | Auto-generated 16-byte salt by `argon2` library |
| Passwords never stored in plaintext | ✅ | Only argon2id hash stored in `passwordHash` field |
| Passwords never logged | ✅ | Raw passwords never appear in audit logs |
| No password length ceiling below 128 chars | ✅ | Max 128 chars (configurable), argon2id has no internal limit |

## Forgot Password

| Requirement | Status | Our Implementation |
|---|---|---|
| Return consistent message whether user exists | ✅ | Always: "If an account exists, we've sent a reset email" |
| Token is cryptographically random | ✅ | `crypto.randomBytes(32)` — 256 bits entropy |
| Token stored as hash in DB | ✅ | SHA-256 hash stored; raw token sent to user |
| Token is short-lived | ✅ | Default 15 minutes (configurable) |
| Token is single-use | ✅ | `usedAt` field set after use, checked on verification |
| Previous tokens invalidated when new one requested | ✅ | All existing tokens of same type deleted before new one created |
| Reset link sent via secure channel (email) | ✅ | Token sent via email (SMTP/Nodemailer) |
| After password reset, revoke all sessions | ✅ | `deleteMany({ userId })` revokes all sessions after reset |

## Input Validation

| Requirement | Status | Our Implementation |
|---|---|---|
| Validate all input server-side | ✅ | Zod schemas validate every request body before controller |
| Use allowlist validation | ✅ | Zod parses known fields, strips unknown fields (`.strip()`) |
| Validate input length (min and max) | ✅ | All string fields have min/max length in Zod schemas |
| Validate input format (email, pattern) | ✅ | `z.string().email()`, `z.string().regex()` |
| Return structured validation errors | ✅ | Field-level error details in standard response format |

## Rate Limiting & Brute Force

| Requirement | Status | Our Implementation |
|---|---|---|
| Rate limit login endpoint | ✅ | Default: 5 attempts per 15 minutes per IP |
| Rate limit registration endpoint | ✅ | Default: 3 attempts per 60 minutes per IP |
| Rate limit forgot-password endpoint | ✅ | Default: 3 attempts per 15 minutes per IP |
| Optional account lockout | ✅ | Lock after N failed attempts for M minutes (configurable, switch-based) |
| Return 429 with Retry-After header | ✅ | `express-rate-limit` sends 429 with standard headers |
